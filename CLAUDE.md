# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Browser prototype of a mahjong-style word solitaire game. TypeScript + React (Vite), plain DOM/CSS rendering, native HTML5 drag-and-drop. Desktop only. See `DESIGN.md` for the full game design — it is the source of truth for rules and should be re-read before changing mechanics.

## Commands

- `npm run dev` — start Vite dev server.
- `npm run build` — type-check (`tsc`) then build (`vite build`). Use this to validate TypeScript; there is no separate lint or test script.
- `npm run preview` — preview the production build.

No tests exist. There is no linter. Verify changes by running the dev server and exercising the affected mechanic.

## Architecture

### State model (single reducer)

The whole game runs through one reducer in `src/game/reducer.ts`. The top-level shape is `AppState = { state: GameState, history: GameState[], outcome, lastError }`. Every successful move pushes the prior `GameState` onto `history`; `ROLLBACK` pops it. `RESET` rebuilds via `buildInitialState` in `src/game/init.ts`. Outcome (`playing` | `won` | `lost`) is recomputed from `GameState` after each move and gates further actions in the reducer; the App also shows a blocking `Overlay` when not `playing`.

Move semantics live in `src/game/moves.ts` (`applyAction` dispatches over the `Action` union, throwing on invalid moves; the reducer catches and stores the message in `lastError`). All move-cost accounting is centralized there.

Types are in `src/types.ts`. `Action` is the set of in-game moves; `AppAction` adds meta actions (`ROLLBACK`, `RESET`, `ADD_BONUS_SLOT`).

### Board geometry (half-tile grid)

The board uses a half-tile coordinate system — this is the most subtle part of the codebase. Cards have integer `(x, y, z)` where `x, y` are in half-card units (a card's footprint is a 2×2 block of cells) and `z` is an explicit layer. **There is no `halfOffset` flag** — odd-parity coordinates are implicitly half-offset.

Interactivity is governed by a single rule in `src/game/coverage.ts`:

**Coverage** (`isSlotRevealed`): a slot is covered if any overlapping-footprint slot has a higher *effective layer* `z * 100 + (stackDepth - 1)`. The `*100` weighting means a same-position-stacked card always counts as visually higher than a same-`z` half-offset neighbour — this matches the CSS `z-index` used in `Board.tsx`. Covered cards render face-down and are non-interactive. Any change to coverage logic must stay consistent with the rendering math in `Board.tsx` (`HALF_W`, `HALF_H`, `LAYER_LIFT`, `STACK_VISUAL_OFFSET_Y`, and the `effectiveLayer` formula). `isSlotRevealed` takes a structural `CoverageSlot` (`{ x, y, cards: {z}[] }`) that `BoardSlot` satisfies, so the editor reuses the exact same rule (see the reveal-preview toggle below) rather than re-deriving it. Reveal ignores any "current layer" notion — many z-layers can be face-up at once (a level may use `z` as a near-global stack/draw order, with most cards alone at their own z), which is why the editor's layer view can blank a card that actually plays face-up.

### Board slots and stacks

A `BoardSlot` is one `(x, y)` position holding a vertical stack of `BoardCardEntry { card, z }`. Only the top of a stack is interactive. Stacks shrink only — cards can be removed from the top (sent to a category slot), but **board slots never accept new drops at runtime**. Level data may declare same-`(x,y)` cards with different `z` (authored stacking), and those cards peel off one at a time as the player clears the layers above. Once a slot empties it stays empty for the rest of the level.

### Category slots

`CategorySlot { lockedCategory, displayedCard, cardsConsumed }`. Empty slot accepts only category cards (locks the slot); occupied slot accepts only matching-category simple cards (consumes them, increments `cardsConsumed`). When all simple cards of the locked category have been consumed across the level (`consumedSimple` filtered by category), the slot auto-clears back to empty. This auto-clear happens inside `placeCardInCategorySlot` in `moves.ts` and is global, not per-slot — multiple slots locked to the same category share the count via the level-wide consumed list.

`ADD_BONUS_SLOT` appends a one-time extra empty slot; `bonusSlotUsed` gates it.

### Win/loss

`isWon`: all simple cards across all categories have been consumed (`consumedSimple.length >= totalSimpleInLevel`). `isLost`: `movesUsed >= movesLimit` and not won. `movesLimit < 0` means unlimited.

### UI layout

Four stacked containers rendered by `App.tsx`: `Header` (stock / hand / moves), `CategorySlotsRow`, `Board`, `Footer` (level picker, highlight toggle, rollback). All components are dumb — they read `GameState` and dispatch `AppAction`. Drag-and-drop uses the native HTML5 API with a custom MIME type via `src/components/dragData.ts` (`hand` or `board` source kinds).

The `highlightUnplayable` UI toggle dims cards with no legal destination ("stranded"). When on, stranded cards are also non-draggable (see `Board.tsx` and `Header.tsx`). Since the board never accepts drops, every drop goes to a `CategorySlot` via `CategorySlotsRow`.

### Levels

Levels are JSON in `src/levels/`, picked up by `src/levels/index.ts` via `import.meta.glob('./*.json', { eager: true })` and ordered by **play order** (see *Play order is canonical* below; filename is only a tiebreak for levels absent from the order). Adding a level: drop a new `*.json` file in the folder — no code edit needed. Format documented in `DESIGN.md`. A `cardId` resolves first against `categoryId` (becomes a category card) then against any `wordId` (becomes a simple card of that category) — see `createCardFromId` in `src/game/cards.ts`. Card UIDs are reset per level load via `resetUidForLevel`.

`src/levels/` here is a **prototype sandbox** — the shipping levels live in the sibling Unity project at `…/SoliJong/unity/soli-jong/Assets/Code/Common/Feature/SoliJong/Levels/` (identical `LevelData` JSON, one `*.json.meta` Unity file alongside each). **Never commit an authored/shipping level into this (editor) repo** — `src/levels/` is only for transient local solver verification; a finished level belongs solely in the Unity Levels folder above. The CLI tools below take a directory argument so they can analyse/fix that folder directly; they only rewrite existing `*.json` and never touch the `.meta` files.

Words can optionally render as images (`icon: true` + `imageId` referencing `/public/images/<imageId>.png`). For image words the `wordId` is **namespaced equal to the imageId** (`<categoryId>__<token>`) — the card set's picture tokens are bare indices (`01`,`02`,…) shared across every category, and `createCardFromId` resolves a board/stock `cardId` to a category by first-match on `wordId`, so the wordId must be unique level-wide. (Older shipping levels with semantic image words like `wordId:"bike"` predate this and stay valid; only `imageSwap.ts` (the Images tool) generates namespaced image words.) Words may also carry `missing: true` and categories `incomplete: true` — placeholder markers the editor's base-fill tool writes for words it couldn't supply (see below); the game ignores them.

A level may carry an optional top-level `difficulty` string (e.g. `"hard"`), authored to flag intentionally hard levels — typically order-traps that straightforward play softlocks on but a stock reorder can't fix (category cards sit on the board). The game ignores it; the editor surfaces it as a dropdown and preserves it through unfill→fill so saves don't drop the tag. The Unity model (`SoliLevelData`) reads it via `[JsonProperty("difficulty")]`.

### Level editor (`src/editor/`, route `#/editor`)

A separate in-browser authoring tool; never part of gameplay (the build still bundles it). It has its own reducer (`reducer.ts`) over a `SkeletonLevel` — categories as `{ letter, simpleCards, pinnedCategoryId?, pinnedWords? }`, board/stock referenced by letter. `unfill.ts` turns a `LevelData` back into a skeleton; `fillSkeleton` in `fill.ts` turns a skeleton into `LevelData`, resolving categories/words from the catalog. Saving uses the File System Access API (`save.ts`): pick a folder once, then writes go straight to `<folder>/<name>.json` (Chrome/Edge; other browsers blob-download). Whether a card renders as a picture is **dictated by its config data** (`icon`/`imageId` in the level JSON, written by the Images tool), not an editor toggle.

**Canvas view (`BoardCanvas.tsx`).** The board canvas is a single-layer authoring view: it renders the chosen `currentLayer` bright and (with the `ghostBelow` toggle) lower layers as dimmed striped tiles with a **transparent letter** — so a card one layer below the top reads as "hidden" even though in-game it's an uncovered top-of-stack and plays face-up. The **`reveal preview`** toggle (`revealPreview` state, `TOGGLE_REVEAL_PREVIEW`) instead colours every card by its real in-game reveal state — face-up + green outline if uncovered, face-down striped if covered — independent of `currentLayer`. It computes this by grouping skeleton cards into `(x,y)` slots and calling the game's `isSlotRevealed` directly (the shared rule above), so the preview can't drift from gameplay. It's view-only; editing (brush/erase/move) still acts on `currentLayer`. (`validate.ts`'s "Covered at start" count is a second, equivalent copy of the same coverage math — kept separate, not yet delegated to `isSlotRevealed`.)

**Word catalog.** `catalog/words.json` is `[{ categoryId, wordsIds }]` — the lookup library; `pools()` in `fill.ts` caches it (merged with `basics.ts`). `catalog/category_list.json` is an ordered list of category names used to assign categories by index. Category-assignment logic for the batch tools is centralized in `rangeAssign.ts` (`computeAssignments`).

**Image catalog.** `catalog/image_categories.json` is a separate `[{ categoryId, wordsIds }]` list of *image-ready* categories — every wordId is the bare picture token of a PNG in `public/images/` (`<categoryId>__<token>.png`). The current set was imported from a fresh card-icon art delivery: **369 categories / ~2061 pictures**, tokens are zero-padded indices (`01`,`02`,…) with no semantic word identity (fine — an icon tile renders only the picture, never the word). It is **not** merged into `pools()` (its ids would collide with text categories in `words.json` and corrupt base fill); it backs the Images tool and the per-category image control only. `catalog/images.ts` (`AVAILABLE_IMAGES`) is the flat set of all `<imageId>` keys present in `public/images/`, regenerated from that folder; `fill.ts` checks it to decide whether a word emits `icon`/`imageId`. `imageCatalog.ts` exposes the catalog to the editor (`imageTokens`, `imageCapability`, `imageCatsWithAtLeast`).

**Importing a new art set (`scripts/import-card-icons.ts`).** Full-replace importer for a fresh card-icon delivery laid out as `<anySubfolder>/<CATEGORY>-<NN>.png` (category name + per-picture index; names recur across folders). It normalizes each category via `toSnake`, **merges by categoryId across folders and dedups identical pictures by content hash**, assigns tokens `01…NN`, then (with `--write`) wipes `public/images/`, copies the renamed PNGs, and rewrites `catalog/image_categories.json` + `catalog/images.ts`. `npx tsx scripts/import-card-icons.ts <srcDir> [--write]` — dry-run by default (prints category count, per-category picture distribution, deduped count, id collisions). Because the new tokens are indices (not semantic words), the old "auto-image when a catalog word happens to have a PNG" path no longer fires; all imaging now goes through explicit image-category assignment (the Images tool).

**Batch own-imaging (`scripts/apply-own-images.ts`).** CLI equivalent of the Images tool's "Use own (all)" applied to every level in a folder: for each text (or stale) category whose **normalized name** has an image set with ≥ its word count, it rewrites that category to namespaced picture cards (`<cat>__<NN>`) and remaps board/stock — **own theme only, never a random one**; already-good image categories and categories without an own set are left untouched, and collisions are skipped. `npx tsx scripts/apply-own-images.ts <levelsDir> [--write]` (dry-run by default; re-serializes as 2-space JSON, no trailing newline; touches only `*.json`, not `*.json.meta`). NOTE: the Unity `Levels/` dir **is git-tracked** (repo: `…/SoliJong/unity/soli-jong`; the parent `SoliJong/` folder is not a repo, which misled earlier notes) — commit or stash before `--write` so a bad run is easy to revert.

**Coverage report (`scripts/image-coverage-report.ts <levelsDir> [out.md]`).** Reports per category slot whether it can be its own pictures: ✓ enough / 🖼 already image / ⚠ too few / ✗ no art; lists the **gaps** (category names used with no art — candidates to generate art for or replace), **partials** (art exists but too few, with the deficit), per-level breakdown, and **unused art categories** (art-backed names no level uses — fill from these). Writes markdown + prints a summary. (For the current 50 Unity levels the level category names and the art set are nearly disjoint — only ~10/373 slots imageable, see [[card-icons-art-swap]].)

**Drop-one-to-fit (`scripts/drop-one-stock-card.ts <levelsDir> [--write]`).** For categories short of their own art by **exactly one** picture, drops one word so it fits — but only a word whose cards are **entirely in the stock** (zero board cards), so board layout is untouched; removes the word from `wordsData` and its stock entries. Run `apply-own-images.ts --write` after to image the now-fitting categories. Dry-run by default.

**AI word generation (dev only).** A Vite middleware in `vite.config.ts` exposes `POST /__editor/generate-words`; it shells out to the local `claude` CLI (`--json-schema`, Haiku) — no API key, only exists under `vite dev`. Generated words are appended to `catalog/words.json` on disk *and* merged into the in-memory pool (`fill.addCatalogWords`); `wordGen.ts` is the client. Vite is configured to ignore `words.json` writes so growing the library mid-session doesn't trigger a reload. The prompt/schema/model live in **`src/editor/wordGenPrompt.ts`** (`buildWordGenPrompt`), shared with the refill CLI so the two can't drift — each request passes the category's existing words as **positive sense/type anchors** (not a bare exclusion list), and the prompt demands new words be **members/types** of the category, never parts/accessories/actions merely *associated* with it (e.g. for "Ship" → Schooner/Corvette, never Anchor/Sail). `rangeAssign.buildGenRequests` populates the `existing` anchors for the Fix tool's Generate button.

**Solver & analyzers (`src/editor/solver/`).** Read-only checks run in a Web Worker (`solver.worker.ts`, debounced via `useSolver`) over the live skeleton and surface as chips in the canvas controls. (1) **Optimal solver** (`solverCore.ts`): A* over `GameState` using the real `applyAction`/`enumerateMoves`; reports min moves to win, or unsolvable. (2) **Difficulty** (`difficulty.ts`): BFS over decision states reporting a `failureHorizon` — the shallowest decision depth where a wrong choice kills the level. (3) **Straightforward analyzer** (`greedy.ts`, `GreedyChip`): simulates a no-lookahead player — feed matching simples first, then lock the **most-feedable** category (the one with the most reachable simples right now, board card or drawn card), and only **lock a category card on sight** (drawn card, or a board card to uncover) when nothing is immediately feedable. Softlock is detected by a `hashState` revisit on a full draw cycle. (Locking the most-feedable category rather than the first by board position is what stops a human-winnable level from being mis-flagged.) When the optimal solver wins but this one softlocks, the level is an **"order trap"** — stock category-card order lures the player into filling every slot before a needed category surfaces. A greedy win also proves solvability (its line is a valid solution), so it doubles as a cheap solvability witness when A\* times out.

**Stock draw order.** `applyDraw` (`moves.ts`) draws the **last** element of the stock array first; a held hand card returns to the front, so the deck recycles. Nothing reverses stock — `level.stock[i]`, `skel.stock[i]` and `GameState.stock[i]` are the same index (so one permutation applies to all three). The editor stock strip shows array order, first-drawn on the right.

**Tools menu — four tools, all write in place to the bound folder:**
- *Fill levels from list* (`BatchFillModal` / `batchFill.ts`): base fill. Walks all level files in order with a category-index cursor that accumulates over **every** valid level (so a level always starts at the index implied by all levels before it, regardless of selection); checkboxes + a write range choose which to (re)write. A category that can't supply enough words is written anyway, padding each missing card slot with a placeholder. No AI, no category replacement here.
- *Fix levels* (`BatchFixModal` / `batchFix.ts`): finds levels with missing words and resolves them per level — generate the missing words (AI) or replace a category with a random catalog one that has enough — then saves. Operates on each level's existing categories (no index reassignment). **Image-aware:** a category whose every card renders as a picture (`wordsData.every(w => w.icon)`) is treated as already filled — its picture tokens aren't in the text catalog, so it is never counted as a shortfall/gap, never offered for generate/replace, and on save its category + every board/stock cardId is spliced back verbatim (`fillFixRow`) instead of re-rolled to text words. So fixing a level's text gaps no longer reverts its imaged categories. The preview *is* the save preview — the words shown are exactly `fillFixRow`'s output (`pinnedWords: p.chosen`), and any word not already in the file is flagged green with a `＋` (`fileRealWords`/`range-word.fills`), so you can see which real words will land in place of each `(needs word N)` stub before saving. **Save all changed** (footer) bulk-writes every level where a save would clear placeholders cleanly (`gapCount === 0 && fileIncomplete`) or apply a category override (`willChange`), skipping unchanged and still-short levels. **Headless CLI counterpart — `scripts/fix-missing-words.ts <levelsDir> [--write] [--seed=N]`:** for every category that still carries `(needs word N)` stubs, it **fills** from the catalog where possible (theme kept) or **replaces** the category with a random valid catalog category (≥ needed words, unique across the whole level set so no theme repeats; `--seed` makes the roll reproducible). Image categories and complete text categories are spliced back **verbatim** (only the filled/replaced slots are rewritten, so the diff is exactly the fix), and every result is validated so categories/words never cross (no word equals a category name, no word shared across categories, every board/stock cardId resolves) — it aborts a slot rather than emit a crossing. Dry-run by default (prints the per-level fill/replace plan); 2-space JSON, no trailing newline, touches only `*.json`. Used to clear the last SoliJong levels the editor's catalog couldn't fill.
- *Images* (`ImagesModal` / `imageSwap.ts`): per-level, swaps chosen categories to image-ready ones (from `image_categories.json`) so the tiles render pictures — **this (writing `icon`/`imageId` into the level) is the only way a card becomes a picture; there is no editor toggle.** Does **not** go through `fillSkeleton` — `imageSwap.ts` (`buildSwapResult`) surgically rewrites `LevelData`, mapping each slot's distinct words 1:1 onto an image category's namespaced tokens (`<categoryId>__<NN>`, an **index never the word** — so the same word in two categories never collides) and remapping every board/stock `cardId`, so the placement/match structure (including words reused across many tiles) is preserved exactly. **Image↔category association is by category, never by word.** Each slot shows an **own-images availability badge** (`OwnBadge` from `ImageSlot.ownImageCount`): `✓ images: N` (its own theme has enough), `⚠ N < need`, or `— no images`. Matching is **by normalized category name** (`ownImageCategory`/`ownImageTokenCount` → `normId`, so a level's TitleCase `Birds` finds the snake `birds` set). Per-slot actions: **use own** (show as its own pictures; disabled unless `ownImageCount ≥ distinctWords`), **pick…** (searchable `ImageSetPicker` over `imageCatsWithAtLeast(need)` for any image set — manual override; each row previews the set's pictures with the first `need` outlined as "will use" and shows how many slots already use it), **→ words** (roll an image slot back to text words — the `TO_WORDS` sentinel override; `buildSwapResult` re-sources words from the text catalog by the category's normalized name, `rollbackWordCount`, since the original words were dropped on imaging; disabled unless enough text words), **reset**. Bulk **Use own (all)** sets every own-ready category to its own pictures and leaves the rest for manual picking (**no random theme is ever assigned**). The row header shows `ownReadyCount` ("N own-ready"). The modal renders each image slot's actual pictures as thumbnails (`Thumb`; missing PNGs show a `?` box). A non-swapped image slot whose pictures aren't in the current `public/images/` set is flagged **stale** (`ImageSlot.stale`, from `AVAILABLE_IMAGES`) — e.g. left over from a previous art set; **use own**/**pick…** re-image stale slots too. Picks must keep all `cardId`s unique; an unresolved slot (shortfall or id collision) is flagged and blocks save. Overrides reuse `overrideKey`/`overridesForLevel` from `batchFill.ts`, keyed by category letter.
- *Fix draw order* (`ReorderModal` / `batchReorder.ts`): finds **order traps** (straightforward player softlocks, see Solver & analyzers) and rewrites the stock so it no longer does. `reorderFix.ts` (`planStockReorder`) runs a greedy-safe constructive scheduler, then — since that scheduler can corner itself even when many winning orders exist (e.g. all category cards in the stock under a tight slot cap) — falls back to a **seeded verified random search** over stock orderings (`searchBudget`, default `REORDER_SEARCH_BUDGET`; the modal passes a smaller one to stay responsive). Every candidate is **verified by re-running the greedy sim** — a greedy win proves solvability, so no A\* re-check. Only traps with category cards on the board are genuinely unfixable by reorder. The reorder is **lossless**: it permutes the existing `stock` cardIds (never re-`fillSkeleton`), applied via `applyOrderToLevel` / `applyOrderToSkeleton` which share one index permutation. Board-driven traps (a category locked from the board, or its simples buried) are flagged unfixable rather than touched. The same fix is one click from `GreedyChip`'s "Fix order" button on the live skeleton (`APPLY_STOCK_ORDER`, undoable). CLI (both take an optional `[dir]`, default `src/levels`): `scripts/test-greedy.ts` reports the analyzer/reorder verdict per level; `scripts/fix-order.ts [dir] [--write]` batch-applies the reorder in place (dry-run by default, preserves each file's trailing-newline style). Point either at the Unity Levels folder to fix the shipping levels.

**Play order is canonical (`src/levels/order.json`).** The web project mirrors the Unity `LevelOrder/default.json` array of `levelId`s at `src/levels/order.json` (helper: `src/levels/order.ts` → `playOrderRank`). Everything that lists or processes levels orders by it — `src/levels/index.ts` (the bundled `LEVELS` → game picker + non-Chromium editor list) and `src/editor/save.ts`'s `listLevelsInFolder` (→ the editor's Chromium dropdown **and** all four batch tools, including base-fill's category cursor in `batchFill.ts`) — with filename only a tiebreak for levels absent from the order (legacy sandbox files, `Tutorial`). So the editor's Fill tool now walks play order, not filename order (this was the source of the "refilled categories by filename" bug). The mirror stays in lockstep automatically: `order-by-difficulty.ts` and `tune-levels.ts` write `src/levels/order.json` alongside the Unity file on `--write`, and `refill-by-play-order.ts` defaults its `orderFile` arg to it. `order.json` is excluded from `index.ts`'s `./*.json` glob (it is the order array, not a level).

A separate CLI orders the whole level **sequence** into a difficulty curve, and is **safe to re-run as levels are added** — `scripts/order-by-difficulty.ts <levelsDir> [orderFile] [--write] [--from=K] [--skip-invalid] [--curve=NAME]`. It scores each level from the analyzers (`D = 0.5·tightness + 0.3·load + 0.2·length` — tightness = A\* optimal-moves / move-limit, load = categories − slots, length = optimal moves, each min-max normalized; greedy-softlock levels forced to the top band), rank-matches the ranking onto the chosen curve, and rewrites the Unity **level-order array** — `…/SoliJong/unity/soli-jong/Assets/Game/LevelOrder/default.json`, a JSON list of `levelId`s loaded at runtime by `LevelOrderProvider` (`MainLevelTrack`; array position = play order). It **reconciles the order against the level files actually present**: new levels are picked up, deleted ones dropped, and re-running on an unchanged set is a **no-op** (idempotent — same ranking, same curve). Broken / `unsolvable` levels abort the run (fail-fast) unless `--skip-invalid` drops them; duplicate `levelId`s abort. `--curve=` selects `sawtooth` (default — the shipped one: each 5-level cycle ramps easy→hard, no macro drift) | `sawtooth-rising` | `sine-rising`. Dry-run by default (prints sparkline, per-cycle listing, and an added/dropped/changed diff vs the current order); `--write` applies it, preserving the one-id-per-line format. Recomputes A\* (~1–2 min for 50). `DEBUG_RANK=1` dumps the per-level ranking. **Live-safe re-order (`--from=K`):** a full re-order changes which level sits at each play index, so a returning player's saved progress index would point at a different level. `--from=K` freezes play-positions `1..K-1` exactly (byte-identical) and re-orders only `K..end`, with newly-added levels landing in that re-ordered tail; set `K` to "released/reached level count + 1" (`--from=<count+1>` is a pure append). The curve target is on absolute play position, so the frozen prefix + re-ordered tail still read as one continuous sawtooth. Freezing a prefix that references a deleted/invalid level aborts. The authoritative level-order file lives in the shipping Unity project; on `--write` the same array is also mirrored to `src/levels/order.json` (the web copy the editor and game order by).

**Faithful move model (waste pile).** The prototype's `hand` is a single card, but the Unity game's hand is an accumulating **waste pile** (klondike: draws stack up, only the top is playable, you can play *down* the pile, and a stock-empty draw recycles the reversed waste — itself a move). So the single-card greedy/A\* counts above are only *approximate* for the shipping game. `solver/wasteGreedy.ts` (`analyzeWasteGreedySkeleton`) models the real mechanic faithfully, reusing the shared board/chain/coverage/category logic (only draw/recycle + top-of-waste access differ), so its move counts and softlocks match Unity. Two consequences: (1) the real game is more move-efficient (you bank several plays per draw burst), so competent move cost runs **~4 lower (up to ~19)** than the single-card estimate — single-card *overestimates*, which matters when setting move limits; (2) only the **board** can't recycle, so **board-driven traps** — a level that softlocks under *both* the single-card and the waste greedy (a category card pinned under coverage, or its simples buried) — are real, whereas a level that softlocks *only* under the waste greedy is just the stock/waste exposure order tripping the no-lookahead heuristic, not a real trap (the pile fully cycles, so it adds no access restriction).

**Move-limit tuning (`scripts/tune-levels.ts`).** The waste-accurate difficulty optimizer + monetization knob, and the tool that produced the **current shipping order and limits** (it supersedes the single-card ranking in `order-by-difficulty.ts` for accuracy). Ranks by **intrinsic** difficulty (board-trap → top band; else `0.6·(categories−slots) + 0.4·competent-move-cost`; move-budget-independent — no A\*, just the two greedy sims), lays onto the curve, then per 5-cycle applies a tightness **mask** (`easy-tight-easy-tight-tight`: cycle positions 1 & 3 generous, 2/4/5 tightened). A tightened *winnable* level gets `movesLimit = waste competent cost + a spare cushion` (`--spare`, default **0** = competent cost, so any wasted move pops — playtested; +10 was too soft, raise only if telemetry says so); board-driven traps keep their budget (they **auto-pop** — the player burns moves cycling a deck they can't clear); generous positions are untouched. Competent cost = waste-greedy moves (single-card greedy as a safe fallback when the waste greedy can't finish a non-trap level). Dry-run by default (prints order + every old→new `movesLimit` + projected popup rate); `--write <levelsDir> <orderFile>` rewrites the order file (and mirrors it to `src/levels/order.json`) and each changed level's `movesLimit` **surgically** (only the number; formatting preserved). Tuned the 50 shipping levels to ~1 popup per 1.7 levels. It does **not** yet support `--from` (full re-order only); use it pre-launch or add prefix-freeze before tuning a live build.

**Category re-fill by play order (`scripts/refill-by-play-order.ts`).** Base-fill assigns `category_list` categories by a cursor walking levels *in order* — originally filename order, which equalled the old play order, so the player met categories in list order. After a difficulty re-order, play order ≠ filename order and that progression breaks. This CLI re-fills every level's category content with the cursor walking **play order** (the order file) instead: `refill-by-play-order.ts <levelsDir> [orderFile] [--generate] [--write]` (orderFile defaults to `src/levels/order.json`, the web mirror). Structure-preserving (unfill → reassign categories → fill): board/stock layout, slot counts, the tuned `movesLimit`s, difficulty tags and the play order itself are all untouched — only category/word identities change (so it does not disturb the difficulty optimization). **Image-safe:** a category that renders as pictures (`wordsData.every(w => w.icon)`) is kept verbatim — its category data plus every board/stock cardId pointing into it are spliced back unchanged (mirrors `batchFix.ts`'s `fillFixRow`) — and only **text** categories are re-themed and refilled text-only. Image slots still consume their play-position index, so a text slot lands on the same `category_list` entry base-fill would give that position. So re-running `--write` over already-imaged levels no longer reverts their images (the old text-only behaviour did); only the text categories' words change. `--generate` AI-generates words for short categories (shells to the local `claude`, like the dev middleware, growing `catalog/words.json`); the generation prompt passes each category's existing words as **positive sense/specificity anchors** (not a bare exclusion list) and forbids synonyms of present words, so it picks the right sense of a polysemous name (e.g. "Palm" the wrist, not the tree) and does not pad closed sets with synonyms — without that anchoring it drifts. Without `--generate`, gaps pad with `(needs word N)` placeholders. Dry-run by default (reports the play-order assignment + per-category shortfall); `--write` rewrites the level files. Used after every difficulty re-order to re-couple categories to play order.

**Gap markers.** When base-fill is short it pads with placeholder words `(needs word N)` (unique level-wide), flagged `missing: true` on the `WordData` and `incomplete: true` on the `CategoryData`. `fillSkeleton(skel, { padGaps: true })` does the padding; without it, fill throws on a shortfall. A pinned category absent from the catalog is **not** a shortfall error — `pickReal` returns it as an empty category and `fillSkeleton` always pads its words with placeholders (regardless of `padGaps`), so such levels still save and the fix tool resolves them later. The fix tool locates levels by these markers (plus live shortfall against the current catalog) and clears them on save.
